# Decision Log — 2026-03-08

## Task: INFR-US4-A006 — Google OAuth with PKCE

**Requirements:** 1.1.2 (user can sign up with Google OAuth)

---

## Decisions Made

### 1. PKCE state validation is a frontend responsibility

The system design says "The API validates the `state` parameter to prevent CSRF during OAuth." In the PKCE authorization code flow for a SPA, the frontend generates the `state`, stores it in sessionStorage, redirects to Google, then validates the `state` on return. The API endpoint (`POST /v1/auth/google`) only receives `code`, `code_verifier`, and `redirect_uri` — not `state`. The `code_verifier` itself provides cryptographic proof-of-possession for the authorization code, which is the core PKCE security guarantee. State validation against CSRF at the frontend layer is the standard pattern for SPA OAuth; storing state server-side and requiring it in the API callback would require session tracking before authentication (circular dependency).

### 2. No golang.org/x/oauth2 dependency — raw HTTP calls to Google

The auth package implements Google token exchange and userinfo fetching with raw `net/http` calls. This avoids adding a new top-level dependency and keeps the auth package's dependency footprint minimal. The two calls needed (token exchange, userinfo) are simple enough to implement directly.

### 3. CodeExchanger interface defined in auth package at the consumer

Following the project's "interfaces at consumer" convention, `CodeExchanger` is defined in the auth package where it is consumed by the Service. The concrete `GoogleExchanger` struct implements it. In main.go, a `GoogleExchanger` is wired in when Google OAuth is configured.

### 4. GetOrCreateOAuthUser uses INSERT ... ON CONFLICT DO NOTHING then SELECT

Rather than a multi-statement CTE UPSERT (complex with partial indexes), the approach is:
1. Attempt `INSERT INTO users (oauth_provider, oauth_id, email) ... ON CONFLICT (oauth_provider, oauth_id) WHERE ... DO NOTHING RETURNING id, email, created_at`
2. If rows returned → new user created, return it
3. If no rows returned (conflict) → `SELECT id, email, created_at FROM users WHERE oauth_provider = $1 AND oauth_id = $2`

This handles concurrent inserts correctly without complex locking.

### 5. Email conflict: OAuth login returns ErrEmailConflict if email already has password account

If a user registered with email+password and attempts Google OAuth with the same email address, the INSERT will fail on the email UNIQUE constraint (not the oauth index). The repository returns `ErrEmailConflict`. The handler returns 409 Conflict. Account linking is out of scope.

### 6. GoogleOAuthHandler is a separate interface from AuthService

The server package defines `GoogleOAuthHandler` as a separate, focused interface with a single `GoogleCallback` method. `AuthService` remains unchanged. `MountAuth` accepts the auth service for existing routes; `MountGoogleOAuth` (or updating MountAuth) accepts the additional interface. Both are satisfied by `*auth.Service`. Keeping them separate avoids bloating `AuthService` and makes it easier to stub in tests.

### 7. Google OAuth config is optional — skips route registration if unconfigured

If `api.google_oauth.client_id` is empty, the API server does not register the `/v1/auth/google` route. Local development without Google credentials still works for all other auth flows.

## Technical Challenges

### 1. Testable endpoint URLs on GoogleExchanger

The tests use `httptest.Server` to mock the Google token and userinfo endpoints. To make the URLs configurable without exposing them in the public constructor, `tokenURL` and `userInfoURL` were added as unexported fields with constants for the production URLs. The production `NewGoogleExchanger` sets them to the real Google URLs. Tests call a package-internal `newGoogleExchangerForTest` helper that sets the URLs to the httptest server addresses. This keeps the public API clean while making unit testing straightforward.

### 2. sqlmock and INSERT ON CONFLICT DO NOTHING

`go-sqlmock` intercepts `QueryRowContext` calls. The INSERT with `RETURNING` uses `QueryRowContext`. When the ON CONFLICT DO NOTHING fires, `QueryRowContext.Scan` returns `sql.ErrNoRows`. In sqlmock, returning an empty rows set (no rows added) correctly simulates this behavior — the Scan call returns `sql.ErrNoRows` when there are no rows to scan.

### 3. WithGoogleOAuth chosen over functional options

The functional options pattern (common in larger Go APIs) was considered but adds indirection. Since there is only one optional feature group (Google OAuth), a single `WithGoogleOAuth` method on `*Service` is simpler and sufficient. It mutates and returns the service, which allows chaining if needed in the future.
