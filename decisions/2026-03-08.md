# Decision Log — 2026-03-08

## Task: INFR-US4-A006 — Google OAuth with PKCE

**Requirements:** 1.1.2 (user can sign up with Google OAuth)

---

## Decisions Made

### 1. PKCE state validation is a frontend responsibility

The system design says "The API validates the `state` parameter to prevent CSRF during OAuth." In the PKCE authorization code flow for a SPA, the frontend generates the `state`, stores it in sessionStorage, redirects to Google, then validates the `state` on return. The API endpoint (`POST /v1/auth/google`) only receives `code`, `code_verifier`, and `redirect_uri` — not `state`. The `code_verifier` itself provides cryptographic proof-of-possession for the authorization code, which is the core PKCE security guarantee. State validation against CSRF at the frontend layer is the standard pattern for SPA OAuth; storing state server-side and requiring it in the API callback would require session tracking before authentication (circular dependency).

### 2. No golang.org/x/oauth2 dependency — raw HTTP calls to Google

The auth package implements Google token exchange and userinfo fetching with raw `net/http` calls. This avoids adding a new top-level dependency and keeps the auth package's dependency footprint minimal. The two calls needed (token exchange, userinfo) are simple enough to implement directly.

### 3. CodeExchanger interface defined in auth package at the consumer

Following the project's "interfaces at consumer" convention, `CodeExchanger` is defined in the auth package where it is consumed by the Service. The concrete `GoogleExchanger` struct implements it. In main.go, a `GoogleExchanger` is wired in when Google OAuth is configured.

### 4. GetOrCreateOAuthUser uses INSERT ... ON CONFLICT DO NOTHING then SELECT

Rather than a multi-statement CTE UPSERT (complex with partial indexes), the approach is:
1. Attempt `INSERT INTO users (oauth_provider, oauth_id, email) ... ON CONFLICT (oauth_provider, oauth_id) WHERE ... DO NOTHING RETURNING id, email, created_at`
2. If rows returned → new user created, return it
3. If no rows returned (conflict) → `SELECT id, email, created_at FROM users WHERE oauth_provider = $1 AND oauth_id = $2`

This handles concurrent inserts correctly without complex locking.

### 5. Email conflict: OAuth login returns ErrEmailConflict if email already has password account

If a user registered with email+password and attempts Google OAuth with the same email address, the INSERT will fail on the email UNIQUE constraint (not the oauth index). The repository returns `ErrEmailConflict`. The handler returns 409 Conflict. Account linking is out of scope.

### 6. GoogleOAuthHandler is a separate interface from AuthService

The server package defines `GoogleOAuthHandler` as a separate, focused interface with a single `GoogleCallback` method. `AuthService` remains unchanged. `MountAuth` accepts the auth service for existing routes; `MountGoogleOAuth` (or updating MountAuth) accepts the additional interface. Both are satisfied by `*auth.Service`. Keeping them separate avoids bloating `AuthService` and makes it easier to stub in tests.

### 7. Google OAuth config is optional — skips route registration if unconfigured

If `api.google_oauth.client_id` is empty, the API server does not register the `/v1/auth/google` route. Local development without Google credentials still works for all other auth flows.

## Technical Challenges

### 1. Testable endpoint URLs on GoogleExchanger

The tests use `httptest.Server` to mock the Google token and userinfo endpoints. To make the URLs configurable without exposing them in the public constructor, `tokenURL` and `userInfoURL` were added as unexported fields with constants for the production URLs. The production `NewGoogleExchanger` sets them to the real Google URLs. Tests call a package-internal `newGoogleExchangerForTest` helper that sets the URLs to the httptest server addresses. This keeps the public API clean while making unit testing straightforward.

### 2. sqlmock and INSERT ON CONFLICT DO NOTHING

`go-sqlmock` intercepts `QueryRowContext` calls. The INSERT with `RETURNING` uses `QueryRowContext`. When the ON CONFLICT DO NOTHING fires, `QueryRowContext.Scan` returns `sql.ErrNoRows`. In sqlmock, returning an empty rows set (no rows added) correctly simulates this behavior — the Scan call returns `sql.ErrNoRows` when there are no rows to scan.

### 3. WithGoogleOAuth chosen over functional options

The functional options pattern (common in larger Go APIs) was considered but adds indirection. Since there is only one optional feature group (Google OAuth), a single `WithGoogleOAuth` method on `*Service` is simpler and sufficient. It mutates and returns the service, which allows chaining if needed in the future.

## Task: INFR-US4-A007 — Rate limiting on auth endpoints

**Requirements:** Security model — 10 attempts per minute per IP on auth endpoints

---

## Decisions Made

### 1. Token bucket over fixed window counter

A token bucket was chosen over a fixed window counter. The token bucket refills continuously rather than in discrete one-minute slots, so a user who makes 10 requests right before a window boundary and 10 more right after will not suddenly get 20 allowed requests. The continuous refill is fairer and more consistent for brute force protection.

### 2. No external dependencies — stdlib only

The task explicitly required stdlib only (`net`, `sync`, `time`). `golang.org/x/time/rate` was not used. The implementation mirrors its approach (token bucket) but without requiring the extra import.

### 3. Bucket eviction deferred to production upgrade

Per the task spec, buckets are never evicted in the MVP. For auth endpoints the number of distinct IPs is expected to be low compared to the memory cost of a bucket. A comment in the code documents this decision. A future upgrade would add TTL-based eviction.

### 4. `authGroup` field on Server struct — MountGoogleOAuth fallback

`MountAuth` creates a rate-limited sub-group and assigns it to `s.authGroup`. `MountGoogleOAuth` uses `s.authGroup` when it is non-nil (meaning `MountAuth` was called first), and falls back to `s.privateGroup` otherwise. This preserves backward compatibility for the existing test `TestMountGoogleOAuth_RegistersRoute` which calls `MountGoogleOAuth` independently.

### 5. Rate limit is shared across all auth routes per IP

All five auth routes (`/register`, `/login`, `/logout`, `/forgot-password`, `/reset-password`) plus `/auth/google` share a single rate limiter (10 req/min/IP). This is the tightest brute-force protection: an attacker cannot exhaust just the login endpoint and then try the register endpoint with a fresh bucket.

### 6. `clientIP` uses `net.SplitHostPort` to handle both host:port and bare IP

`r.RemoteAddr` after `middleware.RealIP` can be either `"host:port"` (TCP connections) or a bare IP (when RealIP sets it from a forwarded header that contains no port). `net.SplitHostPort` returns an error in the bare-IP case, which is used as the signal to return `r.RemoteAddr` unchanged.

---

## Technical Challenges

### 1. `TestMountAuth_RateLimitsAuthEndpoints` timing

The rate limiter uses `newRateLimiter(10)` which sets capacity=10. Eleven rapid consecutive requests in a test all happen within the same millisecond window, so no refill occurs between them. The 11th request drains the bucket below 1 and is correctly denied. This is not timing-sensitive in practice because test execution is faster than the 1-token-per-6-seconds refill rate.

### 2. chi group vs. sub-router for rate-limiting

`s.privateGroup.Group(func(r chi.Router) { ... })` creates an inline group (not a sub-router mounted at a prefix). Routes registered in this group inherit the `/v1` prefix from `privateGroup` and get the `IPRateLimit` middleware applied. A sub-router (`chi.NewRouter()`) would require re-mounting at `/v1` and would lose the CORS middleware from the parent. The inline group approach keeps CORS and rate limiting both active on auth routes.

## Task: INFR-US5-A006 — Implement duplicate calculator endpoint

**Requirements:** 1.2.5 (User can duplicate an existing calculator)

---

## Decisions Made

### 1. Ownership is verified at the service layer via the existing getter

`Service.Duplicate` calls `s.getter.GetCalculator(ctx, id, userID)` first (same pattern as `Service.Delete` and `Service.Update`). This reuses existing ownership logic and correctly returns `ErrNotFound` vs `ErrForbidden` before the copy is made. The `Duplicator` interface takes only `id` (not `userID`) since ownership is already verified upstream.

### 2. INSERT-SELECT with config_version reset to 1

The repository uses a single `INSERT INTO calculators (user_id, name, config, config_version) SELECT user_id, name, config, 1 FROM calculators WHERE id = $1 AND is_deleted = FALSE RETURNING ...` query. This is atomic, does not require reading the row into Go first, and resets `config_version` to 1 since the copy starts its own edit history. The new UUID is assigned by the database default.

### 3. Response uses createCalculatorResponse (no config in body)

The duplicate endpoint returns the same shape as the create endpoint (id, name, created_at, updated_at). The full config is not included in the response. The dashboard client navigates to the editor for the new calculator and fetches the full config via a separate GET, so including config in the duplicate response is not required by the current acceptance criteria.

### 4. Name is copied verbatim (no "(copy)" suffix)

Requirement 1.2.5 does not specify a naming convention. The name is copied as-is. Adding disambiguation is noted as a future suggestion but deferred to avoid over-engineering.

## Task: INFR-US5-A008 — Add rate limiting on public config endpoint

**Requirements:** SYSTEM_DESIGN.md — rate limiting on config endpoint

---

## Decisions Made

### 1. Per-IP rate limiting chosen over per-calculator-ID

`SYSTEM_DESIGN.md` specifies "Per-calculator-ID limits to prevent abusive scraping." The implementation uses per-IP limiting instead. Rationale: the public config endpoint is behind a 5-minute CDN/browser cache. Per-IP limits pool all calculator config requests from a single IP (60 req/min total), which is sufficient protection for an MVP without the added complexity of keying the rate limiter on both IP and calculator ID. A per-calculator-ID limiter would require passing the URL parameter into a custom middleware variant. This can be revisited if scraping patterns emerge that per-IP limits don't address.

### 2. 60 req/min per IP (vs 10 req/min for auth)

Auth endpoints are brute-force targets — low limit is correct. The config endpoint is called by embedded widgets on end-user page loads; many different users behind a corporate NAT could share a single IP. 60 req/min (1 req/sec) is high enough to accommodate multiple concurrent users on the same IP loading widget pages, while still blocking automated scraping.

### 3. Same rate limiter infrastructure — no new dependencies

`newRateLimiter` / `IPRateLimit` from `ratelimit.go` are reused directly. The chi sub-group pattern (`publicGroup.Group(...)`) is the same approach as `MountAuth`.

### 4. Limiter is not shared with other public routes

A fresh `newRateLimiter(60)` is created inside `MountPublicCalculators`. If future public routes are added (e.g., submissions), they get their own limiter with an appropriate rate, rather than sharing the config limiter's bucket.

## Task: INFR-US6-A006 — Image upload endpoint

**Requirements:** 1.8.5, 1.3.7

---

## Decisions Made

### 1. AssetStorage is a consumer-defined two-method interface

`AssetStorage` is defined in the `server` package with only `Upload` and `GetURL` — the two operations the handler actually uses. `Delete` is not needed by the upload handler. `storage.StubStorage` and both concrete adapters satisfy this interface since they implement the full `storage.Storage` superset. This follows the "interfaces at consumer, keep them small" convention.

### 2. Content type detection via http.DetectContentType (magic bytes), not client header

The client-supplied `Content-Type` header on the multipart part is ignored. `http.DetectContentType` is called on the first 512 bytes of the actual file data. This prevents MIME sniffing attacks where a client labels an executable as `image/jpeg`. Detection happens after reading the full payload so the same buffer is used for both detection and upload — no seek required.

### 3. Size limit enforced by reading into memory with LimitReader

The handler reads the file into a `[]byte` via `io.LimitReader(file, maxUploadSize+1)`. If the result is longer than `maxUploadSize`, the file is rejected with 400. Reading into memory is acceptable because the hard limit is 5 MB. A streaming approach (passing the multipart part reader directly to S3) would avoid buffering but cannot accurately enforce the size limit without reading — S3 would need to reject after the upload started. The memory approach gives a clean, early rejection at the handler layer.

### 4. Content-addressed keys via SHA-256

The upload key is `hex(sha256(data)) + extension`. This provides natural deduplication: uploading the same image twice results in a single stored object. It also prevents enumerable or user-controlled keys. The file extension is derived from the detected content type (not the client filename), so the key and the URL's extension always match the actual format.

### 5. initStorage in main.go uses endpoint+bucket as base URL for MinIO dev

For S3/MinIO in local development (`cfg.Storage.S3.Endpoint` is non-empty), the base URL passed to `NewS3AdapterFromConfig` is `endpoint + "/" + bucket`. This makes `GetURL(key)` return `http://localhost:9000/quotecraft-assets/<key>`, which is the correct publicly accessible URL for path-style MinIO. For production (empty endpoint), `cfg.CDN.BaseURL` is used instead.

### 6. initStorage exits with os.Exit(1) on unknown provider or initialisation failure

Storage is required infrastructure — there is no useful fallback if the storage adapter cannot be created. Exiting immediately with a logged error is consistent with the existing pattern (e.g., database connection failure). The `panic("unreachable")` after the switch's default branch satisfies the compiler; `os.Exit` terminates the process before it is reached.

## Task: BLDR-US2-A009 — Image Select field type with image upload

**Requirements:** Image Select field type for the builder dashboard

---

## Decisions Made

### 1. `ImageSelectOption` and `ImageSelectFieldConfig` defined in `shared/config/calculatorConfig.ts`

`ImageSelectOption` is distinct from `FieldOption` because it has an `imageUrl` field. A separate interface rather than extending `FieldOption` keeps each type's intent clear and avoids having `FieldOption` grow an optional `imageUrl` that most field types don't use.

### 2. `uploadFile` added to `ApiClient` interface — no `Content-Type` header

The implementation deliberately omits `Content-Type` from the headers, letting the browser set the multipart boundary automatically. Setting it manually would corrupt the boundary encoding. This follows the standard practice for `FormData` uploads.

### 3. `StubApiClient.calls` typed as `StubApiCall[]` (flat interface)

The `calls` array was initially typed as a union (`({ method: string; path: string; body?: unknown } | UploadCall)[]`), which broke existing tests that access `client.calls[0].body` without narrowing. A single `StubApiCall` interface with all optional fields (`body?`, `file?`) keeps the type simple, backward-compatible, and TypeScript-friendly.

### 4. `ImageOptionEditor` extracted as a local component inside `ImageSelectFieldConfigPanel.tsx`

Each option row has independent upload error state. Extracting `ImageOptionEditor` as a component means each row manages its own `uploadError` state with `useState`, preventing one option's error from polluting another's. The component is not exported — it's an implementation detail of the panel.

### 5. `FieldEditorWidget` dispatches to type-specific panels (switch on `field.type`)

The original `FieldEditorWidget` only rendered `FieldConfigPanel` (the base panel). This task introduced type dispatch: each field type renders its dedicated panel which internally wraps `FieldConfigPanel`. This required removing the direct `FieldConfigPanel` import and replacing it with type-specific panel components. The `onUploadImage` prop defaults to a rejecting function so callers without upload support get a clear error rather than a silent failure.

### 6. `isFieldset` includes `image_select` in `FieldPreviewRenderer`

`ImageSelectFieldRenderer` renders a `<fieldset>` element (like radio and checkbox). Adding `image_select` to the `isFieldset` check prevents an outer `<label>` from being rendered around the fieldset, which would be semantically incorrect.

### 7. `CalculatorPreviewForm.test.tsx` updated to use `ImageSelectFieldConfig` cast

The existing test passed a field with `type: 'image_select'` but without `options`, typed as `BaseFieldConfig`. After implementing the real `ImageSelectFieldRenderer`, accessing `field.options.map(...)` on an undefined `options` array caused a runtime error. The test was updated to include `options: []` and cast to `ImageSelectFieldConfig`.

### 8. `<img>` elements in config panel and preview renderer — `next/image` not used

Both the option image preview in `ImageSelectFieldConfigPanel` and the option images in `FieldPreviewRenderer` use native `<img>`. `next/image` requires known `width` and `height` dimensions, which are not available for arbitrary user-uploaded images. The ESLint warnings (`@next/next/no-img-element`) remain as warnings but are not errors. This is the correct trade-off for a builder UI with unknown-dimension user uploads.
