# Decisions — 2026-03-11

## Task: STYL-US1-A001 — Build color picker controls for primary, secondary, background, and text colors

**Requirements:** 1.7.1

### Decisions Made

**1. Used native `<input type="color">` for color pickers**

No third-party color picker library was introduced. The native `type="color"` input provides browser-level enforcement of hex color values, is accessible, keyboard-operable, and requires zero dependencies. The browser constrains the value to a valid lowercase 6-digit hex string, so no client-side validation is needed for the picker itself.

**2. Added `ThemeConfig` and `DEFAULT_THEME` to `dashboard/src/shared/config/calculatorConfig.ts`**

The theme config lives alongside the other editor config types in `calculatorConfig.ts` rather than in a separate file, consistent with the existing pattern (`LayoutMode`, `Step`, etc. all live in that same file). `theme` is a required field on `CalculatorEditorConfig` — this is intentional since the editor always has a theme state initialized from `DEFAULT_THEME`.

**3. `theme` is a required field (not optional) on `CalculatorEditorConfig`**

Making it required ensures the API always receives theme data on save, and prevents callers from accidentally omitting it. Default values are applied at initialization in `EditorPage` using `DEFAULT_THEME`. Existing test files that constructed `CalculatorEditorConfig` objects without `theme` were updated to include it.

**4. Default colors are Tailwind-aligned hex values**

- `primaryColor: '#3B82F6'` (Tailwind blue-500)
- `secondaryColor: '#6B7280'` (Tailwind gray-500)
- `backgroundColor: '#FFFFFF'` (white)
- `textColor: '#111827'` (Tailwind gray-900)

These are sensible defaults that work well together and are recognizable to developers.

**5. `ColorPickerPanel` placed at the bottom of the left editor column**

The component is mounted after the `OutputList`/`FormulaInput` section in `EditorPage`. This is a temporary placement — STYL-US1-A007 (live preview wiring) will likely lead to a layout review. The component is intentionally placed where it can be found visually without requiring any layout redesign in this task.

**6. Live preview wiring deferred to STYL-US1-A007**

`CalculatorPreviewForm` does not yet accept a `theme` prop. The `theme` state in `EditorPage` is tracked and saved via auto-save, but not yet passed to the preview. That is the scope of A007.

### Technical Challenges

Adding `theme` as a required field to `CalculatorEditorConfig` caused TypeScript errors in two existing test files (`useAutoSave.test.ts` and `calculators.test.ts`) that constructed config objects without `theme`. Both were updated to include `theme: DEFAULT_THEME`. No production code was affected.

jsdom normalizes `<input type="color">` values to lowercase hex (e.g., `#FF0000` → `#ff0000`). Test assertions use lowercase values to match jsdom's behavior.
