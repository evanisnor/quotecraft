# Decision Log — 2026-03-02

## Task: CALC-US1-A009 — Write comprehensive unit tests for edge cases

**Requirements:** 1.5.1 (arithmetic), 1.5.2 (parentheses), 1.5.3 (IF conditional), 1.5.4 (comparison operators), 1.5.5 (variable references)

---

## Decisions Made

### 1. Tests added to existing `evaluate.test.ts` rather than a separate file

The edge case tests augment the evaluator's existing test file rather than creating a new `edge-cases.test.ts`. The existing file already owns the evaluate function's behavioral contract; the new tests extend that contract with more complex inputs. Splitting into a second file would fragment coverage without a meaningful organizational benefit.

### 2. Five new `describe` blocks appended at the end of the outer `describe('evaluate', ...)` block

The new sections are:
- `division by zero — extended edge cases`
- `IF short-circuit evaluation`
- `deeply nested IF expressions`
- `deep nesting of parentheses and unary operators`
- `complex pricing formula patterns`

Each section has a focused scope. This keeps the test file navigable as the formula engine grows.

### 3. `Number.isNaN()` used for all NaN assertions

`result.value === NaN` is always false in JavaScript. All NaN checks use `Number.isNaN(result.value)`. This is the same pattern as the pre-existing division-by-zero tests.

### 4. Short-circuit evaluation tested with a positive control

The short-circuit tests include a case where the taken branch references an unknown variable — confirming that errors in the evaluated branch are correctly surfaced. Without this positive control, tests that simply pass because the branch is skipped could also pass if errors were silently suppressed globally.

### 5. Multi-assertion `it()` blocks split after code review

The first code review flagged multi-assertion `it()` blocks as a convention violation (per SKILL.md). The following were fixed:
- IF-as-condition split into two `it()` blocks (truthy/falsy paths)
- Percentage discount split into two `it()` blocks
- 3-tier pricing converted to a `test.each` table
- Fractional comparison split into two `it()` blocks
- Not-equal condition split into two `it()` blocks

---

## Technical Challenges

### Division-by-zero NaN test initially used the wrong input

An initial test for `{a} / {b}` with `a=10, b=0` expected NaN, but `10 / 0 = Infinity` in JavaScript. Only `0 / 0` produces NaN. The test was corrected to use `a=0, b=0` for the NaN case and `a=7, b=0` for the Infinity case.

## Task: BLDR-US4-A001 — Build preview pane container with style isolation

**Requirements:** Builder dashboard — calculator preview panel

---

## Decisions Made

### 1. Shadow root guard uses `host.shadowRoot` check, not a ref flag

React StrictMode mounts effects twice in development. Rather than tracking attachment with a boolean ref, the guard reads `host.shadowRoot` directly — if it's already set, the shadow root was already attached and we just surface it via `setShadowRoot`. This is safe because `attachShadow` throws a `DOMException` if called on a host that already has a shadow root, so the guard is both necessary and the idiomatic check.

### 2. CSS reset injected via `document.createElement('style')` inside the effect

Tailwind cannot reach into shadow DOM, so all reset styling must be injected directly into the shadow root. The style element is created imperatively inside the `useEffect` before setting state. This ensures the reset is in place before React renders any children into the shadow root via `createPortal`.

### 3. `createPortal(children, shadowRoot)` gated on `shadowRoot !== null`

The shadow root is only available after the effect runs (post-mount). Gating the `createPortal` call on a non-null state value means children are never rendered into an undefined target. On the first render (before the effect), the host div is empty.

### 4. Two-column layout in `EditorPage` uses Tailwind flex utilities only

Layout is expressed with `className="flex gap-6"` and `className="flex-1"` — no inline styles. Left column holds all existing editor controls, right column holds `<PreviewPane />`. The `PreviewPane` is empty for now; children will be wired in A002–A004.

### 5. `waitFor` used in EditorPage tests for preview pane assertions

The `PreviewPane` section element renders synchronously but shadow DOM attachment is async (useEffect). The new EditorPage tests use `waitFor` for the region assertion to avoid flakiness if the test runner checks before the effect completes.

---

## Technical Challenges

### jsdom 26 supports `attachShadow` — no polyfill needed

Earlier jsdom versions did not support shadow DOM. jsdom 26 (used by jest-environment-jsdom 30) supports `attachShadow` natively. Verified this before writing tests so no mock or polyfill of `attachShadow` was needed.

### `within` helper does not reach into shadow DOM

As noted in the task brief, `@testing-library/react`'s `within` helper cannot traverse into shadow roots. The children test queries `host.shadowRoot!.querySelector(...)` directly, which is the correct approach for shadow DOM assertions in jsdom.

## Task: BLDR-US4-A002 — Wire shared field renderers into the preview

**Requirements:** 1.3.11 (live preview panel), 1.5.9 (formula engine integration)

---

## Decisions Made

### 1. `FieldPreviewRenderer` dispatches via a `switch` on `field.type` with internal sub-components

Each field type is handled by a private sub-component (`NumberFieldRenderer`, `SliderFieldRenderer`, etc.) rather than inlining all JSX into a single large switch. This keeps each renderer independently readable and testable without making internal components part of the public API.

### 2. `radio` and `checkbox` fields use `<fieldset>` + `<legend>` for label, not a standalone `<label>`

`<label htmlFor>` requires a single target element. Radio groups and checkbox groups have multiple inputs, so the accessible pattern is `<fieldset><legend>...</legend></fieldset>`. The outer `FieldPreviewRenderer` wrapper suppresses the `<label>` for these two types (`isFieldset` flag).

### 3. Checkbox checked state is tracked locally in `CheckboxFieldRenderer`, not derived from the numeric `value` prop

The `value` prop represents the count of checked options. Reversing that count back to individual checked states is not possible (which options were checked is not encoded in the count). Local `useState<Set<string>>` tracks which option IDs are checked. `onChange` is called with `next.size` after each change. This matches the spec: "value is count of checked options."

### 4. `TextFieldRenderer` owns local `displayValue` state; always calls `onChange(0)`

Text fields are not formula-usable yet. The component tracks the typed string for display only and always emits `0` to the parent. This avoids complicating `FormulaContext` with string values (which the formula engine doesn't support).

### 5. `CalculatorPreviewForm` merges field defaults with user values via `useMemo`

The form separates concerns: `userValues` (state) tracks values the user has explicitly entered; `values` (memoized) merges `buildFieldDefaults(fields)` over `userValues`. New fields get their defaults when added. Existing user inputs are preserved because `userValues` spreads over the defaults for those keys. Using `useMemo` avoids the `react-hooks/set-state-in-effect` lint rule that would have triggered with a `useEffect` + `setState` approach, and avoids a cascading render that `useEffect` introduces.

### 6. Number input test uses `fireEvent.change` instead of `userEvent.type`

`FieldPreviewRenderer` is a controlled component: the `value` prop comes from the parent, and the parent in the test is not stateful (onChange is a mock). After `user.clear`, the value stays at the controlled value `5` (mock doesn't update state), so subsequent `user.type('42')` appends to `5` yielding `542`. `fireEvent.change` with `{ target: { value: '42' } }` directly fires the `onChange` event with a specific value, bypassing this uncontrolled/controlled discrepancy. This is the correct testing approach for reading what value was emitted from an event handler without needing a stateful wrapper.

---

## Technical Challenges

### `user.clear` + `user.type` on a controlled number input doesn't behave as expected

Since `FieldPreviewRenderer` is controlled (value flows in as a prop) and the test uses a non-stateful onChange mock, the input's displayed value doesn't update when the mock is called. This caused `user.type('42')` to produce `54` and `52` instead of `42` (typing appended to the pre-existing controlled value `5`). Fixed by switching to `fireEvent.change` for that specific test.

### `image_select` has no `BaseFieldConfig` subtype

The `FieldType` union includes `image_select` but there is no `ImageSelectFieldConfig` interface in `calculatorConfig.ts`. The renderer handles this case with a simple `<p>` placeholder, requiring no type assertion or additional interface.

## Task: BLDR-US4-A003 — Wire shared formula engine into the preview

**Requirements:** 1.5.9 (formula engine integration into live preview)

---

## Decisions Made

### 1. `ResultOutputConfig` added to `calculatorConfig.ts` in the shared config segment

The type lives alongside the other field configuration types (`BaseFieldConfig`, `NumberFieldConfig`, etc.) in `dashboard/src/shared/config/calculatorConfig.ts` and is exported from `dashboard/src/shared/config/index.ts`. This mirrors the approach used for all other calculator config types and keeps output configuration co-located with input field configuration.

### 2. Formula engine wired via `useMemo`, not `useEffect`

Results are computed as derived state during render: `useMemo` maps each `ResultOutputConfig` to a `FormulaResult` by calling `evaluate(output.expression, values)`. Using `useEffect` + `setState` would introduce a cascading render and would violate the SKILL.md principle of not using effects for data transformation. Since `evaluate` is a pure function with no side effects or I/O, `useMemo` is the correct React primitive.

### 3. `outputs` prop defaults to `[]` in the component signature

The `CalculatorPreviewForm` component uses a default parameter (`outputs = []`) so callers that do not supply outputs get sensible behavior without special-casing at call sites. This keeps the EditorPage usage explicit (`outputs={[]}`) while avoiding null/undefined guards inside the component.

### 4. Results section only renders when `outputs.length > 0`

The `<section aria-label="Results">` is conditionally rendered using `{outputs.length > 0 && ...}`. The requirement is that no Results section appears when outputs is empty or undefined; checking the length after defaulting to `[]` covers both cases with a single guard.

### 5. Errors rendered with `role="alert"` inside the results section

When `result.error` is defined, the error message is wrapped in `<span role="alert">`. This provides the required accessibility signal and satisfies the acceptance criteria without introducing a separate notification system.

### 6. Jest `moduleNameMapper` added to resolve `@quotecraft/formula-engine` in tests

The formula engine is a workspace package (`packages/formula-engine`). Jest cannot resolve workspace packages via the `node_modules` symlink the same way Next.js/tsc can, so a `moduleNameMapper` entry is added to `dashboard/jest.config.ts` pointing directly to the package's source `index.ts`.

---

## Technical Challenges

### Formula engine uses `{variableName}` syntax, not bare identifiers

Initial test expressions used bare `quantity * 3` syntax, which the formula engine tokenizer rejects (identifiers without braces are function names or reserved keywords, not variable references). The correct syntax is `{quantity} * 3`. Tests were updated after observing the error message: "Unknown identifier 'quantity': bare identifiers are not allowed; use variable references (e.g. {quantity})."

### Exhaustiveness guard added to `getInitialValue` after code review

Code review flagged the `getInitialValue` switch on `field.type` as having no `default` case. A future `FieldType` addition would cause a silent `undefined` return where `number` is expected. A `default` branch with a `never` type assertion was added so TypeScript will produce a compile error if the switch becomes non-exhaustive.
