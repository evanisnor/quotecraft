# Decision Log — 2026-03-07

## Tasks: BLDR-US5-A001, BLDR-US5-A002, BLDR-US5-A003 — Save & Auto-Save

**Requirements:** 1.2.3 (update calculator config — auto-save is the UX implementation of config persistence)

---

## Decisions Made

### 1. Editor route page converted to client component following dashboard page pattern

The editor route (`app/(dashboard)/editor/[id]/page.tsx`) was originally an async server component. Since the auth token is stored in `localStorage` (client-side only), the `ApiClient` cannot be created server-side. The dashboard page already uses the pattern of being a `'use client'` component that reads `useAuthState()`, creates the `ApiClient` via `useMemo`, and passes it as a prop to the view component. The editor route was updated to follow the same pattern using `useParams<{ id: string }>()` instead of the server-side `params` prop.

### 2. `ApiClient` passed as a prop to `EditorPage`, not created inside the component

`EditorPage` receives `client: ApiClient` as a required prop rather than calling `useAuthState()` internally. This keeps the view component testable without an `AuthProvider` wrapper (tests pass a `StubApiClient` directly), follows the existing `DashboardPage` convention, and separates the concern of client creation (route page) from the concern of rendering the editor (view component).

### 3. `CalculatorEditorConfig` type defined in `shared/config/calculatorConfig.ts`

The config payload sent to the API (`{ fields, outputs }`) is represented as `CalculatorEditorConfig`. It lives alongside the other calculator config types in the shared config module. Future config properties (styling, layout, multi-step) will extend this type as their respective stories are implemented. Adding a separate file was not warranted at this stage.

### 4. `useAutoSave` hook uses refs for stable `performSave` callback

The `performSave` callback is created once with `useCallback([], [])` and reads config, calculatorId, and client via refs rather than as dependencies. This avoids recreating `performSave` on every render (which would reset the debounce timer via the effect dependency array). Refs are kept in sync with the latest values via dedicated `useEffect` hooks. This is the standard pattern for stale-closure avoidance with async callbacks in React.

### 5. First render is explicitly skipped to prevent an immediate save on mount

`useAutoSave` tracks whether it is the first render via a `useRef<boolean>` initialized to `true`. The `useEffect` that schedules the debounce skips on first render and sets the ref to `false`. This ensures that loading an existing calculator (which will later fetch and populate config) does not immediately PUT the empty default config over the stored one. It also prevents spurious saves when the component mounts.

### 6. Retry uses exponential backoff: 1s, 2s, 4s (up to 3 retries, 4 total attempts)

The initial attempt plus 3 retries matches the widget's config-fetch retry convention (per SYSTEM_DESIGN.md). Backoff delays are `1000 * 2^(attempt-1)` ms. This is implemented as a `while` loop inside `performSave` using `await new Promise(resolve => setTimeout(resolve, delay))`. The `isSaving` ref prevents concurrent invocations.

### 7. `'use client'` directive only on boundary components, not on feature-layer hooks/components

Code review flagged `'use client'` in `useAutoSave.ts` and `SaveStatusIndicator.tsx`. The directive belongs only at the component boundary that crosses into Next.js client rendering (`EditorPage.tsx` already has it). Feature-layer code should be framework-agnostic. Directives removed from both files.

### 8. `SaveStatusIndicator` uses `role="status"` + `aria-live="polite"` for screen reader support

The status container uses `role="status"` so screen readers announce status changes without interrupting the user. The `role="alert"` on the persistent error message uses the more assertive announcement appropriate for an error condition. This follows ARIA live region best practices.

### 9. `EditorPage` tests pre-enqueue 20 success responses in `renderEditor` helper

Code review identified that tests rendering `EditorPage` with a `StubApiClient` containing no queued responses would throw `"StubApiClient: no more queued responses"` if auto-save fired (after the 2s debounce). Although the debounce makes this unlikely during normal test execution (tests complete well under 2s), pre-enqueueing success responses makes the tests robust against timing variations and fake-timer usage.

---

## Technical Challenges

### `jest.runAllTimers()` insufficient for testing sequential retry backoffs

Initial retry tests used `jest.runAllTimers()` after advancing time for the debounce. This ran the first backoff timer but not the subsequent ones created by each failed retry attempt, because `runAllTimers()` does not re-run after new timers are created by the executed callbacks. The tests reported `status === 'saving'` (stuck mid-retry) instead of the expected terminal state.

Fixed by switching to `jest.runAllTimersAsync()` (available since Jest 29.5, project uses Jest 30), which repeatedly flushes timers and microtasks in a loop until no pending timers remain — correctly handling the chain of sequentially-created backoff timers.

## Task: CALC-US2-A002 — Formula Text Input with Inline Validation

**Requirements:** 1.5.9 (formula editor shows live preview / inline validation)

---

## Decisions Made

### 1. Validation uses `tokenize` + `parse`, not `evaluate`

The component validates the expression by running it through the tokenizer and parser only. `evaluate` is not called because no field context is available at edit time. This correctly separates syntax validation (at-edit) from runtime errors (unknown variables, division by zero), which only surface during evaluation in the preview pane.

### 2. Unknown variable references are NOT a parse-time error

The formula parser treats `{variable_name}` syntax as valid identifiers. Variable resolution is an evaluate-time concern. This design decision means the builder gets no red underline when referencing a variable name that doesn't exist yet — consistent with how formula editors (e.g., spreadsheets) typically work.

### 3. `fieldVariableNames` prop is accepted but unused

The prop is included in the component's interface and accepted with an `_` prefix (to suppress the unused-variable lint warning) with a JSDoc comment. It's reserved for future enhancement (e.g., autocomplete or variable-existence warnings). Accepting it now avoids a breaking API change later.

### 4. `FormulaInput` does not carry `'use client'`

Feature-layer components don't carry `'use client'` — this is the same decision applied to `SaveStatusIndicator`. The `'use client'` boundary is at the view layer (`EditorPage.tsx`), which already covers all feature components rendered within it.

### 5. Branch coverage of the generic `catch` fallback tested via `jest.requireMock`

The `@quotecraft/formula-engine` package exports non-configurable properties (ESM module semantics), so `jest.spyOn` fails with "Cannot redefine property". The workaround is to declare `jest.mock('@quotecraft/formula-engine', ...)` at the top of the test file with `jest.requireActual` for all exports. This makes the mock object mutable, allowing `jest.requireMock` to override `parse` for a single test.

## Task: CALC-US2-A003 — Live Result Preview Using Current Field Defaults

**Requirements:** 1.5.9 (formula editor shows live preview)

---

## Decisions Made

### 1. `buildFieldDefaults` extracted to `shared/lib/fieldDefaults.ts`

The helper that maps fields to default numeric values was private to `CalculatorPreviewForm.tsx`. Both the preview widget and the formula editor need it. Rather than duplicating the logic, it was extracted to `shared/lib`. This is consistent with FSD: shared utilities live in the `shared` layer and are available to all upper layers.

### 2. `FormulaInput` computes preview via `evaluate`, not by caching AST

`validateExpression` calls `tokenize` + `parse` to check syntax. When valid, the component then calls `evaluate(expression, fieldValues ?? {})` which internally tokenizes and parses again. This is a duplicate parse pass but is acceptable: formula expressions are short, and the formula engine is fast. Introducing AST caching would add complexity for no perceptible gain.

### 3. Preview is rendered as `<output>` with `aria-label`, not `role="alert"`

The computed result is informational — it updates on every keystroke but does not represent an error or urgent state. Using `<output>` with `aria-label="Formula result preview"` is semantically appropriate (HTML `<output>` is designed for computed values). The existing parse error continues to use `role="alert"`, which is correct for error conditions.

### 4. Evaluation errors shown in the preview area, not as alerts

When `evaluate` returns an error (e.g., unknown variable, NaN), the error is shown inside the same `<output>` element as a plain text message (`Evaluation error: {message}`). This keeps the UX clean: one place for the formula result/state, one place for syntax errors.

### 5. `fieldVariableNames` prop annotated with CALC-US1-A008 reference

The prop was added in the previous task as a placeholder for future "did you mean?" variable suggestions. Code review flagged it as dead code without a tracking reference. A comment was added to the prop declaration identifying CALC-US1-A008 as the task that will implement the behavior, resolving the ambiguity without removing the prop.

### 6. Mock mutation test hardened with `try/finally`

The existing test that mutated `engine.parse` inline and restored it after `render()` was fragile — if `render` threw, the restore would not run. Wrapped in `try/finally` so the original is always restored, preventing test contamination.

## Task: INFR-US4-A005 — Password Reset Flow (Forgot + Reset Endpoints)

**Requirements:** Referenced from INFR-US4 (authentication infrastructure)

---

## Decisions Made

### 1. `ErrResetTokenNotFound` defined as a separate sentinel from `ErrInvalidResetToken`

The repository layer signals that a token row was absent via `ErrResetTokenNotFound`. The service layer maps this to `ErrInvalidResetToken` before returning to callers — the same way `ErrSessionNotFound` is mapped to `ErrInvalidSession`. This keeps the repository's vocabulary internal (infrastructure concern) while the service exposes a single domain-level error for callers (invalid or expired token). The HTTP handler only needs to know about `ErrInvalidResetToken`.

### 2. ForgotPassword always returns 200 OK even when email is unknown

The HTTP handler returns `200 OK` regardless of whether the email is registered. The service already returns `nil` for unknown emails (no enumeration). The handler does not check whether an email was sent — it simply calls `ForgotPassword` and returns 200 on nil error or 500 on unexpected error. This prevents user enumeration through the HTTP response.

### 3. `PostgresUserRepository` implements `UserPasswordUpdater` directly

Rather than creating a separate `PostgresUserPasswordRepository`, `UpdateUserPassword` was added as a method on the existing `PostgresUserRepository`. The users table owns the password_hash column, so this is cohesive. In main.go, `userRepo` is passed as both `UserReader` and `UserPasswordUpdater`, which is the same pattern already used with `sessionRepo` for the three session interfaces.

### 4. `LogPasswordResetEmailSender` placed in `internal/auth/email.go`

The email sender implementation is in the auth package alongside the interfaces it implements. This avoids creating a new package for a single development-mode struct. It is clearly documented as a dev-only implementation and should be replaced with a real email adapter (SMTP, SES, etc.) when the email infrastructure story is implemented.

### 5. `resetPasswordHandler` returns 204 No Content on success

Consistent with `logoutHandler`, which also returns 204 for a state-mutating action with no response body. The password has been updated and the token consumed — there is no data to return.

### 6. `newServiceForTest` updated to accept all 10 dependencies

Rather than providing two constructors or optional parameters, `newServiceForTest` was extended to accept all new dependencies in parallel with `NewService`. Existing tests were updated to use `defaultStubs()` + `newServiceForTest`, which is explicit and maintainable. The `defaultStubs()` helper function in the test file returns all 10 stubs at once, reducing boilerplate in individual test functions.

---

## Technical Challenges

### Existing tests used `NewService` with 5 parameters

All existing auth service tests called `NewService(users, userReader, sessions, deleter, reader)`. After the signature change to add 5 new parameters, these calls became compile errors. The tests were updated to use `newTestService()` (which calls `NewService` with all 10 default stubs) or to call `NewService` directly with `defaultStubs()` destructuring. The approach chosen for each test was based on which stubs needed customization.
