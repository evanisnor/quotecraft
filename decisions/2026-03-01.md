# Decision Log — 2026-03-01

## Task: BLDR-US2-A003 — Implement Dropdown field type with option list editor

**Requirement:** 1.3.1 — Builder supports Dropdown / Select fields with configurable options (label + value pairs)

---

## Decisions Made

### 1. `FieldOption` and `DropdownFieldConfig` placed in `shared/config/calculatorConfig.ts`

These types belong in `shared/config` because they are domain types shared across feature boundaries. `OptionListEditor` will be reused by A004 (Radio) and A005 (Checkbox), and those features also need `FieldOption`. Placing the types in `shared/config` prevents lateral imports between feature slices.

### 2. `generateId()` placed in `shared/lib/generateId.ts`

The ID generator is a stateless utility with no domain knowledge — it belongs in `shared/lib`. It is intentionally not exposed to tests so tests do not assert on ID format, only on the presence and type of the generated ID string.

### 3. `OptionListEditor` is deliberately unaware of field type

The component accepts `FieldOption[]` and `onChange: (options: FieldOption[]) => void` with no coupling to dropdown, radio, or checkbox. This allows A004 and A005 to reuse it directly without modification.

### 4. Index-based input IDs (`option-label-0`, `option-value-0`)

The task spec calls for index-based IDs since positions are stable within a single interaction. The `key` prop uses `option.id` (stable across re-renders) while the `htmlFor`/`id` pairing uses the array index so labels always associate with the correct input even after options are removed.

### 5. Disambiguation in `DropdownFieldConfigPanel` tests

When `DropdownFieldConfigPanel` renders both `FieldConfigPanel` (which has a "Label" label for `#field-label`) and `OptionListEditor` (which has "Label" labels for `#option-label-0`, `#option-label-1`, etc.), `getByLabelText('Label')` is ambiguous. Tests that target the base field label input use `getByLabelText('Label', { selector: '#field-label' })` to resolve the ambiguity without changing any production code.

### 6. `DropdownFieldConfigPanel.handleBaseUpdate` explicitly reattaches `type: 'dropdown'`

`FieldConfigPanel` receives and returns `BaseFieldConfig`, which only knows `type: FieldType`. When the base panel calls `onUpdate`, the returned object loses the `'dropdown'` discriminant. `handleBaseUpdate` spreads the base result and explicitly sets `type: 'dropdown'` alongside the existing `options` array to restore the `DropdownFieldConfig` shape.

---

## Technical Challenges

### Label ambiguity in `DropdownFieldConfigPanel` tests

When rendering both panels together, there are multiple elements labeled "Label" (one from the base panel's field label input, two from option label inputs). The fix was to qualify the base panel's label lookup with `{ selector: '#field-label' }` — the ID is fixed and stable in `FieldConfigPanel`.

No production code was changed to work around this; it was purely a test query precision issue.

## Bugfix: Next.js root path returning 404 (unrelated to any task)

### Problem

Running `make dev` in the `dashboard` package produced a 404 for `GET /` immediately on startup. All App Router routes were unreachable.

### Root cause

`next.config.ts` had a custom `pageExtensions: ['page.tsx', 'page.ts', 'page.jsx', 'page.js']` setting. The intent was to prevent Next.js from treating FSD barrel files in `src/pages/` (e.g. `src/pages/dashboard/index.ts`) as Pages Router routes.

The setting had the opposite effect on App Router files. Next.js constructs the expected filename for each App Router special file as `{special-name}.{extension}`. With the custom extensions, it looked for `page.page.tsx`, `layout.page.tsx`, etc. — none of which exist. Every route returned 404 as a result.

### Resolution

Two changes:

1. **Removed `pageExtensions` from `next.config.ts`**, restoring the default `['tsx', 'ts', 'jsx', 'js']`. This allows `page.tsx` and `layout.tsx` to be recognized as App Router special files again.

2. **Renamed `src/pages/` to `src/views/`**. With default extensions, Next.js would scan `src/pages/` and pick up FSD barrel files (e.g. `src/pages/dashboard/index.ts`) as Pages Router routes. Renaming the directory removes it from Next.js's scan path entirely without any framework configuration.

Import paths in the three App Router route files that referenced `@/pages/*` were updated to `@/views/*`.

## Task: BLDR-US2-A010 — Implement drag-and-drop field reordering

**Requirement:** 1.3.9 — Fields can be reordered via drag-and-drop

---

## Decisions Made

### useRef vs useState for draggingIndex
Used `useRef<number | null>` instead of `useState` for the active drag index. `handleDragOver` fires at high frequency; reading stale `useState` closures caused incorrect splice indices during rapid drags. `useRef.current` is read/written synchronously, eliminating the stale-closure bug. A separate `draggingId` state (`useState<string | null>`) drives the `data-dragging` attribute — it tracks the dragged item by its stable `field.id` so the visual indicator follows the item across reorders.

### if (e.dataTransfer) guard retained
`React.DragEvent<T>.dataTransfer` is typed as non-nullable, making the guard look like dead code. However, jsdom's `fireEvent.dragStart` does not populate `dataTransfer`, so the guard is required for test compatibility. A clarifying comment was added to explain this.

### Keyboard accessibility added
ArrowUp/ArrowDown reordering added via `onKeyDown` handler on each `<li>`, with `tabIndex={0}` for focusability. This goes beyond the literal drag-and-drop requirement but is needed to make the feature keyboard-accessible. The `handleKeyDown` swap uses destructuring assignment.

### onReorder fires on each dragOver, not on drop
Reorder is committed eagerly on every `dragOver` event rather than only on `drop`. This gives live feedback to the parent during drag and matches the live-preview requirement (1.3.11). The `handleDrop` handler only calls `e.preventDefault()` to suppress browser default drop behavior.

## Tasks: INFR-US2-A006 (db-seed) and INFR-US2-A007 (db-reset)

**Requirements:** Local development infrastructure — database seeding and reset tooling.

---

## Decisions Made

### 1. `hasher` injected as a function parameter in `run()`

`bcrypt.GenerateFromPassword` is passed as a `func([]byte, int) ([]byte, error)` argument to `run()`. This keeps the function pure for testing — tests substitute `fakeHasher` (deterministic, no bcrypt cost) and `errorHasher` (always fails) without needing any interface wrapping. The `main()` function passes `bcrypt.GenerateFromPassword` directly. This matches the task spec exactly and avoids over-engineering.

### 2. `UserSeeder` interface defined in the `main` package

The interface is defined where it is consumed (in `cmd/db-seed`) per the project's hexagonal architecture rule: interfaces belong at the consumer, not the implementer. There is no value in placing this interface in an internal package since `postgresSeeder` is only ever used by this command.

### 3. `postgresSeeder.SeedUser` wraps the email in the error message

The error from `db.ExecContext` is wrapped with the user's email so that if a seed fails in production, the log message identifies which user triggered the failure without requiring log correlation. The `ON CONFLICT (email) DO NOTHING` clause means conflicts are silently skipped — only genuine database errors propagate.

### 4. `stubSeeder` uses a 1-based `errOnCall` index

The test stub triggers an error on the Nth call (1-based). This makes it easy to express "fail on the second user" without additional state machinery. An `errOnCall` of 0 means never fail, matching the zero value of `int`.

### 5. A007 is pure Makefile composition — no new Go code

`db-reset` chains `migrate down -all`, `migrate up`, and `$(MAKE) db-seed`. Using `$(MAKE)` instead of a direct `go run` invocation ensures `DATABASE_URL` is forwarded correctly to the seed step and respects any future changes to the `db-seed` target without duplication.

### 6. `bcrypt.DefaultCost` (10) used in `run()` as the cost constant

The task spec says cost 10. `bcrypt.DefaultCost` is defined as 10 in the standard library, so it is used directly — makes the intent readable without a magic number.

---

## Technical Challenges

None. Both `github.com/lib/pq` and `golang.org/x/crypto` were already in `go.mod`, so no dependency changes were needed.

## Task: CALC-US1-A001 — Implement tokenizer/lexer for formula expressions

**Requirements:** 1.5.1, 1.5.2, 1.5.4, 1.5.5, 1.5.10, 1.5.11, 1.5.12, 1.5.13 (tokenizer layer)

---

## Decisions Made

### Token design: string union, not enum
`TokenKind` is a string literal union rather than a TypeScript enum. This avoids enum runtime overhead and is more tree-shakeable — relevant since the formula-engine is shared with the widget bundle.

### IDENT covers all keywords — classification deferred to parser
`IF`, `MIN`, `MAX`, `ABS`, `ROUND` all tokenize as `IDENT`. The distinction between keywords and user-defined identifiers is the parser's responsibility. The tokenizer has no concept of reserved words.

### Two-character operator lookahead ordering
`!=`, `>=`, `<=` are matched before their single-character prefixes to prevent the first character from being consumed prematurely. The `twoChar` lookahead string is computed on every iteration; this is intentional for clarity over micro-optimisation.

### Edge cases fixed in code review
1. **Trailing decimal (`1.`)**: After consuming `.`, at least one digit must follow — otherwise `TokenizeError`. Without this, `"1."` would produce a token value of `"1."` that is neither an integer nor a valid decimal.
2. **Empty variable name (`{}`)**: After consuming `{`, `pos === start` (no ident chars consumed) throws `TokenizeError` before the closing-brace check.

### Public API exports include tokenizer surface
`tokenize`, `TokenizeError`, `Token`, `TokenKind` are exported from `index.ts`. The parser and any downstream consumer catching `TokenizeError` by type (e.g. `instanceof TokenizeError`) must import from the package root.

### `TokenizeError` prototype chain note (deferred)
For ES5 widget build targets, `Object.setPrototypeOf(this, TokenizeError.prototype)` may be needed in the constructor for correct `instanceof` behaviour. Deferred to WDGT-US1 integration once the widget's tsconfig target is confirmed.

## Task: CALC-US1-A002 — Implement parser that produces a restricted AST

**Requirements:** 1.5.1, 1.5.2, 1.5.4, 1.5.5, 1.5.10, 1.5.11, 1.5.12, 1.5.13 (parser layer)

---

## Decisions Made

### 1. AST nodes are plain interfaces, not classes

All AST node types (`NumberLiteralNode`, `VariableNode`, `BinaryOpNode`, `UnaryOpNode`, `FunctionCallNode`) are plain TypeScript interfaces with `readonly` fields. The discriminated union `ASTNode` ties them together. This keeps AST nodes as pure data — zero runtime cost, no prototype chain, fully serialisable.

### 2. `Parser` is a private class; `parse` is the sole public API

The `Parser` class tracks mutable `pos` state during parsing. Exposing only the `parse` function (not the class) prevents consumers from holding a partially-consumed parser. Each `parse` call gets an independent, single-use `Parser` instance.

### 3. EOF check sits in `parse`, not in `Parser.parseExpression`

After `parseExpression()` returns, the top-level `parse` function calls `parser.peek()` directly to inspect the remaining token. This keeps the grammar rules clean — the EOF constraint is an entry-point concern, not part of the recursive descent grammar itself. The `peek()` method is package-visible (no access modifier in the class definition) to enable this without a separate public method.

### 4. Function-name allow-list implemented as a `Set<string>` + type guard

`FUNCTION_NAMES` is a module-level `Set<string>`. `isFunctionName()` narrows to `FunctionName`. This approach avoids a switch statement or repeated string comparisons in the hot path while remaining tree-shakeable. The set is frozen at module load time.

### 5. Unknown identifier vs unknown function name: two distinct error paths

When the parser sees `IDENT` followed by `(`, it validates the name and throws `ParseError` with a "Unknown function name" message. When it sees `IDENT` not followed by `(`, it throws "Unknown identifier". Both messages include the offending name so errors are actionable.

### 6. Unary minus is right-recursive

`parseUnary` calls itself recursively for the operand, producing right-associative unary chains. `--{x}` parses as `UnaryOp(-, UnaryOp(-, Variable(x)))`. Double negation is syntactically valid; semantic enforcement (e.g. preventing nonsensical negations) is left to the evaluator.

### 7. Comparison operators are left-associative (same as addition)

The grammar rule for comparison is an iterative `while (true)` loop rather than a single binary match. This produces left-associative chains: `1 < 2 < 3` → `BinaryOp(<, BinaryOp(<, 1, 2), 3)`. This matches the spec note that chained comparisons are syntactically valid.

### 8. `parseFloat` for number literal conversion

Token values for numbers are strings produced by the tokenizer (e.g. `"3.14"`). `parseFloat` converts them to JavaScript `number` at parse time. This means the AST stores the numeric value, not the raw string, which simplifies the evaluator.

### 9. No `eval`, `Function`, `window`, or browser globals

The parser operates exclusively on the token array. No dynamic evaluation or browser-global access exists. This satisfies the sandboxing requirement and makes the parser safe to run in any environment (Node, browser, widget).

## Task: CALC-US1-A003 — Implement AST evaluator with arithmetic, comparisons, and parentheses

**Requirements:** 1.5.1, 1.5.2, 1.5.4, 1.5.10, 1.5.11

---

## Decisions Made

### 1. Division by zero follows JavaScript semantics

`n / 0` returns `Infinity` and `0 / 0` returns `NaN`. These are not treated as errors — the evaluator does not add a special check. Rationale: the spec calls out division by zero as an edge case to test in A009 (P0, "Write comprehensive unit tests for edge cases"), so the decision about whether to return an error or JS semantics is deferred to that task. Surfacing `Infinity`/`NaN` via the `value` field is consistent with how a spreadsheet-like engine behaves.

### 2. Unknown variables default to 0 without surfacing an error

When `evalNode` encounters a `VariableNode` whose name is not present in the `context`, it returns `0` (via `context[node.name] ?? 0`). No `error` field is set. Rationale: A005 ("Add variable reference resolution against field value map") and A008 ("Implement error messages with 'did you mean?' suggestions for unknown variables") are the correct homes for error handling on unknown variables. Defaulting to 0 is consistent with spreadsheet engines and avoids breaking formulas that reference fields not yet filled in.

### 3. FunctionCall nodes throw EvaluateError as a deliberate stub

`evalNode` throws `EvaluateError` for any `FunctionCallNode` (`IF`, `MIN`, `MAX`, `ABS`, `ROUND`). The `evaluate` entry point catches this and returns `{ value: 0, error: message }`. This is intentional — `IF` is implemented in A004 and `MIN`/`MAX`/`ABS`/`ROUND` in A006. Throwing rather than silently returning 0 makes it obvious that a function call was encountered, helping catch misconfigured formulas in the builder.

### 4. `evaluate` is the sole public entry point and never throws

The `evaluate` function wraps the entire tokenize → parse → evalNode pipeline in a try/catch. Any `TokenizeError`, `ParseError`, or `EvaluateError` is caught and returned as `{ value: 0, error: message }`. Callers do not need to handle exceptions; they only inspect the `FormulaResult`. This simplifies the formula editor UI — it can always read `result.error` without a try/catch.

### 5. `FormulaResult` extended with `error?: string`

Added an optional `error` field to `FormulaResult` in `types.ts`. The field is absent on success and present with a human-readable message on failure. An alternative was to use a discriminated union `{ ok: true, value } | { ok: false, error }`, but the optional-field approach is simpler and consistent with how formula editors typically expose errors (check if `error` exists, otherwise use `value`).

### 6. Comparisons return 1 (true) or 0 (false)

Comparison operators (`=`, `!=`, `>`, `<`, `>=`, `<=`) return numeric 1 or 0 rather than booleans. This is consistent with the evaluator being `(ast, variables) → number` (a pure numeric function) and enables comparisons to be used inside arithmetic: `{a} > 0` can appear as the condition in an `IF` call or be multiplied in an expression.

---

## Technical Challenges

None significant. The tokenizer and parser were already complete; the evaluator is a straightforward recursive walk of the AST. The `BinaryOp` inner switch is exhaustive over all 11 `BinaryOperator` values — TypeScript's `strict` mode confirms this at compile time.

## Task: CALC-US1-A004 — Add IF/THEN/ELSE conditional expression support

**Requirements:** 1.5.3 (`IF(condition, then_value, else_value)`)

---

## Decisions Made

### 1. Short-circuit (lazy) evaluation for IF branches

The condition is evaluated first. Only the selected branch is evaluated — the other branch is not touched. This is the correct semantic for a conditional: `IF(x = 0, 1/0, 1)` should not attempt to evaluate `1/0` when `x != 0`. It also avoids unnecessary work in deep nested IF trees.

### 2. Truthiness: `condition !== 0`

Any non-zero numeric value (including negative numbers) selects the then-branch; exactly zero selects the else-branch. This is consistent with the evaluator being `(ast, variables) → number` — comparisons already return 1 (true) or 0 (false), so the IF condition naturally integrates with the comparison operators without a separate boolean type.

### 3. Arity validation before evaluation

`args.length !== 3` is checked before any argument is evaluated. A wrong-arity call never triggers partial evaluation. The error message names the expected and actual argument counts with a description of each parameter.

### 4. Dispatch pattern: `if (name === 'IF')` not a switch

A single `if` guard is used rather than a `switch (name)` because only IF is implemented in this task. The remaining names (`MIN`, `MAX`, `ABS`, `ROUND`) fall through to the existing "not yet supported" throw. When A006 implements those functions, the pattern will be converted to a `switch` with a `default: never` exhaustiveness guard (per the code review suggestion).

### 5. `FunctionCall` case wrapped in a block scope

The `case 'FunctionCall'` now uses `{ }` to scope `const { name, args }`. This mirrors the existing `case 'BinaryOp'` block style and prevents `const` declarations from leaking into the surrounding switch.

---

## Technical Challenges

None. The parser already produced `FunctionCallNode` with `name: 'IF'` and `args: [cond, then, else]` — the evaluator just needed to handle the IF case instead of throwing. No AST changes or parser changes were required.
