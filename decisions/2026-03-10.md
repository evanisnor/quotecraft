# Decisions ‚Äî 2026-03-10

## Task: BLDR-US3-A003 ‚Äî Implement progress bar component for multi-step mode

**Requirements:** 1.4.4

### Decisions Made

**1. Placed `ProgressBar` as an internal component of the `calculator-preview` widget**

The `ProgressBar` component lives in `widgets/calculator-preview/ui/ProgressBar.tsx` and is not exported from the widget's `index.ts`. This keeps it as an internal implementation detail of `CalculatorPreviewForm`. If it needs to be reused in the widget runtime later, it can be moved to `shared/ui` at that point.

**2. `isMultiStep` vs `isMultiStepMode` distinction**

Two separate booleans were introduced in `CalculatorPreviewForm`:
- `isMultiStep = layoutMode === 'multi-step' && steps.length > 0` ‚Äî drives whether the progress bar and navigation UI render
- `isMultiStepMode = layoutMode === 'multi-step'` ‚Äî used for empty state logic only

The distinction matters: when in multi-step mode with zero steps, the empty state should appear even though `isMultiStep` is false.

**3. `safeStepIndex` clamping during render**

Rather than using a `useEffect` to synchronize `currentStepIndex` when steps are removed, `safeStepIndex` is derived as `Math.min(currentStepIndex, steps.length - 1)` during each render. Navigation handlers use `safeStepIndex` rather than the raw `currentStepIndex` to compute the next value. This keeps state minimal and avoids unnecessary effect-driven state updates.

**4. Results shown only on the last step in multi-step mode**

`showResults` is `true` in single-page mode (always), and in multi-step mode only on the final step. This follows the natural wizard UX pattern where the result is revealed at the end. Requirements 1.4.4 doesn't specify this, but it follows from the wizard-style flow (1.4.2).

**5. Multi-step navigation wired into `CalculatorPreviewForm` (not a separate feature)**

Navigation (Next/Back) and step-scoped field visibility were included in this task because they are the functional prerequisites for the progress bar being meaningful. Without navigation, the progress bar would always show "Step 1 of N" and be decorative only.

### No Technical Challenges
The implementation was straightforward. All 469 tests pass; E2E tests confirm correct behavior in a live browser.
