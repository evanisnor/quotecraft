# 2026-03-15 Decision Log

## Task: CALC-US1-A006 — Add MIN, MAX, ABS, ROUND function support

**Requirement:** 1.5.13

### Decisions

**Argument count enforcement — evaluator vs. parser**

The parser's `parseArgs()` previously unconditionally called `parseExpression()` before checking for `)`, meaning zero-argument calls like `MIN()` would fail with a confusing "Unexpected token 'RPAREN'" parser error. The evaluator had `args.length < 1` guards for MIN/MAX that were dead code as a result.

Decision: Update `parseArgs()` to return `[]` when the next token is `RPAREN`, allowing zero-arg calls to parse successfully and reach the evaluator's arity checks. This produces domain-appropriate error messages ("MIN requires at least 1 argument") instead of generic parser errors. All five function arity checks (IF, MIN, MAX, ABS, ROUND) are now exercised by tests.

**ROUND decimal precision**

`ROUND(x, n)` uses `Math.round(n)` to coerce the precision to an integer before computing the scale factor. This means `ROUND(3.456, 1.9)` rounds to 2 decimal places (treated as n=2). This behavior is consistent with "round the precision argument to the nearest integer" and avoids errors for fractional precision values.

**MIN/MAX with 1 argument**

MIN and MAX accept 1 or more arguments (not requiring at least 2). A single-argument MIN/MAX is degenerate but not invalid — it returns the single value. This matches Math.min/Math.max semantics and avoids arbitrary restrictions not present in the requirement.

## Task: STYL-US2-A001 — Implement badge component in the shared field renderers

**Requirement:** 1.7.9

### Decisions

**Badge split across `packages/field-renderers` and `dashboard/src/shared/ui/field-renderers/`**

The shared package (`packages/field-renderers`) exports pure constants (`BADGE_TEXT`, `BADGE_HOMEPAGE_URL`) and a `badgeHtml()` function returning an HTML string. No DOM or React — the package stays framework-agnostic so the widget can import it directly when implemented in WDGT-US1. The React `PoweredByBadge` component lives in the dashboard's own `shared/ui/field-renderers/` and uses the constants from the package to build JSX.

**`badgeHtml()` returns a string, not a DOM element**

The test environment for `packages/field-renderers` is `node`, not `jsdom`. Returning an HTML string keeps the shared package testable without a DOM polyfill and is the natural input format for Shadow DOM injection in the widget.

**`<footer>` wrapper for positional guarantee**

The badge is wrapped in a `<footer>` as the last child of `CalculatorPreviewForm`. This provides a semantic anchor ("bottom of the calculator") that is structurally enforced rather than relying on render order, and gives A003's CSS sanitization a predictable container element to reference.

**`className="qc-badge"` on the React component**

Both `badgeHtml()` and `PoweredByBadge` use `qc-badge` as the CSS class, ensuring consistent styling and giving A003 a stable selector in both the widget (HTML string) and the dashboard preview (React component).

**Feature-flag gating deferred to A002**

The badge always renders unconditionally. Visibility control via the config's feature flag is the scope of STYL-US2-A002.
