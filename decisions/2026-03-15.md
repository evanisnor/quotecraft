# 2026-03-15 Decision Log

## Task: CALC-US1-A006 — Add MIN, MAX, ABS, ROUND function support

**Requirement:** 1.5.13

### Decisions

**Argument count enforcement — evaluator vs. parser**

The parser's `parseArgs()` previously unconditionally called `parseExpression()` before checking for `)`, meaning zero-argument calls like `MIN()` would fail with a confusing "Unexpected token 'RPAREN'" parser error. The evaluator had `args.length < 1` guards for MIN/MAX that were dead code as a result.

Decision: Update `parseArgs()` to return `[]` when the next token is `RPAREN`, allowing zero-arg calls to parse successfully and reach the evaluator's arity checks. This produces domain-appropriate error messages ("MIN requires at least 1 argument") instead of generic parser errors. All five function arity checks (IF, MIN, MAX, ABS, ROUND) are now exercised by tests.

**ROUND decimal precision**

`ROUND(x, n)` uses `Math.round(n)` to coerce the precision to an integer before computing the scale factor. This means `ROUND(3.456, 1.9)` rounds to 2 decimal places (treated as n=2). This behavior is consistent with "round the precision argument to the nearest integer" and avoids errors for fractional precision values.

**MIN/MAX with 1 argument**

MIN and MAX accept 1 or more arguments (not requiring at least 2). A single-argument MIN/MAX is degenerate but not invalid — it returns the single value. This matches Math.min/Math.max semantics and avoids arbitrary restrictions not present in the requirement.

## Task: CALC-US1-A007 — Implement execution timeout (100ms abort)

**Requirement:** 1.5.11 (sandboxed formula engine)

### Decisions

**Synchronous deadline via `Date.now()` polling**

JavaScript is single-threaded, so true preemption of synchronous code is impossible. The chosen approach is to record a deadline (`Date.now() + 100`) at the start of `evaluate()` and check `Date.now() > deadline` at the entry of each `evalNode()` recursive call. This ensures timeout detection is proportional to recursion depth, which bounds the maximum CPU time before a check fires for any formula expression.

**Deadline scoped to `evalNode` only (not tokenize/parse)**

The deadline is set before `tokenize` and `parse` are called, but those phases do not check it internally. Both phases operate on bounded string inputs and are CPU-cheap in practice. Including them in the timed window is acceptable since the timer starts before them, but they don't abort mid-execution. This is a minor limitation acceptable for the current task scope.

**`TimeoutError` follows the same error shape as `EvaluateError`**

`TimeoutError` is caught by `evaluate()`'s generic `catch` block and returned as `{ value: 0, error: message }`, consistent with how `EvaluateError`, `TokenizeError`, and `ParseError` are handled. Callers see a uniform `FormulaResult` regardless of failure type.

**`TimeoutError` exported from `index.ts`**

Although `evaluate()` swallows all errors internally, exporting `TimeoutError` allows consumers who want to discriminate error types (e.g., to display a specific "your formula is too complex" message) to do so by checking the error string rather than the type. The string-based discrimination is fine because `evaluate()` never re-throws.

## Task: STYL-US2-A001 — Implement badge component in the shared field renderers

**Requirement:** 1.7.9

### Decisions

**Badge split across `packages/field-renderers` and `dashboard/src/shared/ui/field-renderers/`**

The shared package (`packages/field-renderers`) exports pure constants (`BADGE_TEXT`, `BADGE_HOMEPAGE_URL`) and a `badgeHtml()` function returning an HTML string. No DOM or React — the package stays framework-agnostic so the widget can import it directly when implemented in WDGT-US1. The React `PoweredByBadge` component lives in the dashboard's own `shared/ui/field-renderers/` and uses the constants from the package to build JSX.

**`badgeHtml()` returns a string, not a DOM element**

The test environment for `packages/field-renderers` is `node`, not `jsdom`. Returning an HTML string keeps the shared package testable without a DOM polyfill and is the natural input format for Shadow DOM injection in the widget.

**`<footer>` wrapper for positional guarantee**

The badge is wrapped in a `<footer>` as the last child of `CalculatorPreviewForm`. This provides a semantic anchor ("bottom of the calculator") that is structurally enforced rather than relying on render order, and gives A003's CSS sanitization a predictable container element to reference.

**`className="qc-badge"` on the React component**

Both `badgeHtml()` and `PoweredByBadge` use `qc-badge` as the CSS class, ensuring consistent styling and giving A003 a stable selector in both the widget (HTML string) and the dashboard preview (React component).

**Feature-flag gating deferred to A002**

The badge always renders unconditionally. Visibility control via the config's feature flag is the scope of STYL-US2-A002.

## Task: STYL-US2-A002 — Wire badge visibility to the feature flag in config response

**Requirement:** 1.7.9

### Decisions

**`feature_flags` added to public config response, hardcoded to `false`**

The billing system is not yet implemented. All users are on the free tier, so `branding_removable` is always `false` server-side. The field is established in the response shape now so the widget (WDGT-US1) and the billing implementation can wire to it without changing the API contract.

**`FeatureFlags` type lives in `dashboard/src/shared/config`**

The widget will define its own type when built (WDGT-US1), keeping the shared package framework-agnostic. For now, `FeatureFlags` is only consumed by the dashboard.

**`fetchPublicConfig` in `entities/calculator/api`**

The EditorPage fetches feature flags from the public `/v1/calculators/{id}/config` endpoint on mount using the authenticated `ApiClient` (the auth header is sent but ignored by the unauthenticated endpoint — harmless). This is the only place feature flags are available until a user profile or subscription endpoint is built.

**Error path defaults to free tier**

If `fetchPublicConfig` fails (network error, etc.), the `.catch(() => {})` keeps `featureFlags` at the initial state `{ brandingRemovable: false }`, ensuring the badge stays visible. This is the safe default per requirement 1.7.9 (badge must show on free tier).

## Task: CALC-US1-A008 — Implement error messages with "did you mean?" suggestions

**Requirement:** 1.5.12

### Decisions

**Levenshtein distance implemented inline — no external dependency**

A self-contained iterative two-row DP implementation is added directly to `evaluate.ts`. No npm package is used. The formula engine must remain dependency-free (requirement 1.5.10/1.5.11 — client-side, sandboxed). The two-row approach is memory-efficient: only O(n) space rather than O(m×n).

**Row copy uses element-by-element loop, not spread**

`prev.splice(0, prev.length, ...curr)` was the initial approach but was replaced with an element-by-element `for` loop. The spread form passes all elements as individual function arguments, which can throw a `RangeError` in V8 when the array length exceeds the engine's argument count limit (~65,536). For variable names this limit is never hit in practice, but a loop is unconditionally safe.

**Suggestion threshold: `Math.max(1, Math.floor(name.length / 3))`**

This scales the accepted edit distance with the length of the unknown variable name. For short names (1–2 chars, threshold 1), only a single-edit candidate qualifies. For longer names (e.g., 12 chars, threshold 4), more drift is tolerated. The minimum threshold of 1 ensures that even single-character unknown variables can produce a suggestion when a close candidate exists.

**Tie-breaking by lexicographic order**

When two candidates have the same Levenshtein distance, the lexicographically earlier one is returned. This is deterministic and requires no additional state beyond comparing strings.
