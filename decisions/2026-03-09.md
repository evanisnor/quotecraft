# 2026-03-09 Decision Log

## Task: BLDR-US3-A001
## Requirements: 1.4.1, 1.4.2, 1.4.3

### Summary

Implemented the layout mode toggle (single-page vs multi-step) for the calculator builder.

### Decisions

**Data model: `layoutMode` and `steps` as separate state fields in `CalculatorEditorConfig`**

Added `layoutMode: LayoutMode` and `steps: Step[]` as top-level fields alongside `fields` and `outputs`. A discriminated union (e.g., `LayoutState = { mode: 'single-page' } | { mode: 'multi-step'; steps: Step[] }`) was considered but rejected for this task: the separate-fields approach is simpler and already matches the pattern used by `fields` and `outputs`. The code reviewer flagged the union approach as a suggestion, not a required change.

**Step preservation on single-page toggle-back**

When switching from multi-step back to single-page, `steps` is preserved in state (not cleared). This allows round-tripping back to multi-step mode without losing step assignments. The config payload will contain `layoutMode: 'single-page'` alongside a non-empty `steps` array — this is intentional. Renderers should ignore `steps` when `layoutMode === 'single-page'`.

**Initial step on first multi-step entry**

On the first entry into multi-step mode (`steps.length === 0`), an initial step is created containing all current field IDs. Title is hardcoded to `'Step 1'`. On subsequent entries into multi-step mode, the existing steps are preserved.

**Known gap: step fieldIds sync**

`steps[].fieldIds` is NOT kept synchronized when fields are added or deleted. A field added or deleted while the builder is in multi-step mode will not appear in (or be removed from) any step's `fieldIds`. This is intentionally deferred to BLDR-US3-A002 (step manager).

### Technical Challenges

**TypeScript diagnostic false positives**: The IDE reported stale errors (unused variables, module-not-found) from intermediate file states written by the webdev agent. All errors resolved after completing the implementation and running `tsc --noEmit` (exit 0).

**Test fixture updates**: Adding `layoutMode` and `steps` as required fields on `CalculatorEditorConfig` broke existing test fixtures in `useAutoSave.test.ts` and `calculators.test.ts` that constructed config objects with only `{ fields, outputs }`. Updated all fixtures.

**`body` typed as `unknown` in `StubApiClient`**: Accessing nested properties on `client.calls[0].body` requires destructuring with a type assertion (e.g., `const { config } = body as { config: ... }`) rather than chaining on the unknown value directly.

## Task: BLDR-US3-A002
## Requirements: 1.4.1, 1.4.2, 1.4.3

### Summary

Implemented the StepManager feature component and integrated it into EditorPage, with step/field synchronization fixed.

### Decisions

**Controlled input for step title rename**

The step title `<input>` uses `value` (controlled) so it stays in sync with the parent-driven `steps` prop. In component-level unit tests with mocked callbacks, the controlled value does not update between keystrokes because the parent mock never updates the prop. Using `userEvent.clear` + `userEvent.type` in this scenario results in each typed character being appended to the original frozen value. The test was written with `fireEvent.change` instead, which fires a single change event with the complete target value — the correct approach for testing the callback contract without needing state to flow back through a real parent.

**Uncontrolled select for field assignment**

The assign-to-step `<select>` uses `defaultValue=""` (uncontrolled) with an `onChange` handler that calls `onMoveFieldToStep`. This is intentional: once a field is assigned, the parent removes it from the unassigned list entirely (the select unmounts), so there is no need to reset the select's own state back to the placeholder. Using an uncontrolled select avoids the need for local state just to reset the dropdown after each selection.

**`handleAddField` step sync uses closure over `layoutMode`**

The `handleAddField` handler reads `layoutMode` from the outer closure (not from within a `setSteps` updater function). This is correct because `layoutMode` is a separate state value and its current snapshot is valid at the time the user clicks. There is no race condition because field additions are not batched concurrently with layout mode changes — the user performs one action at a time.

**`handleDelete` captures `selectedFieldId` before clearing it**

The `handleDelete` handler captures `const deletedId = selectedFieldId` before calling `setSelectedFieldId(null)`. This ensures the cleanup `setSteps` call uses the correct ID even though React batches the two `setState` calls. Without the capture, `selectedFieldId` would still be available synchronously in the closure, but the explicit capture makes the intent clear.

### Technical Challenges

**Controlled input rename test with mocked callback**: Initial test used `user.clear` + `user.type` on the controlled step title input. Because the `onRenameStep` mock doesn't update the `steps` prop, the input resets to the original value after each character — so typing "Intro" after clearing produced calls like ("s1", "r"), ("s1", "o"), etc. Switched to `fireEvent.change` with a complete target value to test the callback contract directly, which is the correct approach for controlled inputs in isolated unit tests.
