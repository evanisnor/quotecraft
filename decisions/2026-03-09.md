# 2026-03-09 Decision Log

## Task: BLDR-US3-A001
## Requirements: 1.4.1, 1.4.2, 1.4.3

### Summary

Implemented the layout mode toggle (single-page vs multi-step) for the calculator builder.

### Decisions

**Data model: `layoutMode` and `steps` as separate state fields in `CalculatorEditorConfig`**

Added `layoutMode: LayoutMode` and `steps: Step[]` as top-level fields alongside `fields` and `outputs`. A discriminated union (e.g., `LayoutState = { mode: 'single-page' } | { mode: 'multi-step'; steps: Step[] }`) was considered but rejected for this task: the separate-fields approach is simpler and already matches the pattern used by `fields` and `outputs`. The code reviewer flagged the union approach as a suggestion, not a required change.

**Step preservation on single-page toggle-back**

When switching from multi-step back to single-page, `steps` is preserved in state (not cleared). This allows round-tripping back to multi-step mode without losing step assignments. The config payload will contain `layoutMode: 'single-page'` alongside a non-empty `steps` array — this is intentional. Renderers should ignore `steps` when `layoutMode === 'single-page'`.

**Initial step on first multi-step entry**

On the first entry into multi-step mode (`steps.length === 0`), an initial step is created containing all current field IDs. Title is hardcoded to `'Step 1'`. On subsequent entries into multi-step mode, the existing steps are preserved.

**Known gap: step fieldIds sync**

`steps[].fieldIds` is NOT kept synchronized when fields are added or deleted. A field added or deleted while the builder is in multi-step mode will not appear in (or be removed from) any step's `fieldIds`. This is intentionally deferred to BLDR-US3-A002 (step manager).

### Technical Challenges

**TypeScript diagnostic false positives**: The IDE reported stale errors (unused variables, module-not-found) from intermediate file states written by the webdev agent. All errors resolved after completing the implementation and running `tsc --noEmit` (exit 0).

**Test fixture updates**: Adding `layoutMode` and `steps` as required fields on `CalculatorEditorConfig` broke existing test fixtures in `useAutoSave.test.ts` and `calculators.test.ts` that constructed config objects with only `{ fields, outputs }`. Updated all fixtures.

**`body` typed as `unknown` in `StubApiClient`**: Accessing nested properties on `client.calls[0].body` requires destructuring with a type assertion (e.g., `const { config } = body as { config: ... }`) rather than chaining on the unknown value directly.
